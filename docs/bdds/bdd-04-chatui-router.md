# bdd-04 — chatui / router

## Use case

As an **authenticated user**, when I type an utterance into the `chatui`
composer at `/chatui/`, I see a response driven by the **choosing tier**
(the router): either a typed action descriptor I can act on, or a closed
outcome telling me the router found no match or that the request needs
human escalation. The router never returns free prose ([[CHATBOT]],
[[adr-15-chatbot-two-tier]] rule 1).

## Scenarios

### Happy path — utterance resolves to an action

```gherkin
Given an authenticated user with a session established via Cognito ([[AUTH]])
And the user holds the Django group required for the intent they are about to name
When the user submits an utterance to the chatui composer
Then the router selects exactly one member of the closed, permission-filtered
  enum built server-side before inference ([[adr-15-chatbot-two-tier]] rule 2)
And the response is a typed action descriptor (`navigate` or `confirm`)
And the composer renders the descriptor as an actionable affordance, never as
  restated prose
```

### NO_MATCH — utterance matches no permitted intent

```gherkin
Given an authenticated user
When the submitted utterance does not correspond to any enum member available
  to this user
Then the router returns the closed NO_MATCH outcome
And the composer shows a neutral "no match" state, not an error page
And no actuator fires
```

### ESCALATE — router defers to a human path

```gherkin
Given an authenticated user
When the utterance corresponds to a case the router is designed to defer
  rather than resolve
Then the router returns the closed ESCALATE outcome
And the composer surfaces the escalation path without ever forwarding the
  user's raw text past the router ([[adr-15-chatbot-two-tier]] rule 5)
```

### Unauthenticated redirect

```gherkin
Given a visitor with no established Django session
When they request `/chatui/`
Then they are redirected to the Cognito-backed sign-in flow ([[AUTH]])
And no request reaches the router
```

### Permission-gated intent stays invisible

```gherkin
Given an authenticated user who does NOT hold the Django group required for a
  given intent
When the user submits an utterance that would otherwise match that intent
Then the enum offered to the router never contained that intent — it was
  filtered out server-side, before inference ([[adr-15-chatbot-two-tier]]
  rule 2, [[adr-10-auth]])
And the router can only return NO_MATCH, ESCALATE, or a different,
  authorized action — never the gated one
```

## Frontend half

`/chatui/` is a top-level page ([[GLOSSARY]]: `chatui`), server-rendered by
Astro SSR. The composer is a **Svelte island** — the escalation ladder
([[adr-04-frontend-and-design-system]] rule 3: server HTML → HTMX → Svelte
island) is climbed to the top rung here because the composer owns genuinely
client-side state — the in-progress draft and the in-flight request status —
that HTMX's server-owned-state model does not fit. Rendering the returned
action descriptor (`navigate`/`confirm`) and the closed NO_MATCH/ESCALATE
states is island-local presentation logic; no domain fragment HTML is
generated by Astro or Svelte ([[adr-05-htmx]] rule 2 stays satisfied — the
island never substitutes for a Django-rendered fragment). Design compliance
follows [[DESIGN-SYSTEM]]; only `PUBLIC_*` variables reach this surface
([[VARIABLES]]).

## Backend half

The router endpoint is the [[API]] row that accepts the utterance and
returns the closed outcome (action descriptor, NO_MATCH, or ESCALATE),
authenticated via the standard session ([[AUTH]], [[adr-10-auth]]) and
authorized by filtering the candidate enum through the user's Django groups
before the model is invoked ([[adr-15-chatbot-two-tier]] rules 2–3). Backend
capability detail: [[CHATBOT]]. Each backend increment this BDD entry
requires feeds its own [[TDD]] entry, linked here once written:
[[tdd-03-router-bedrock-inference]] (real Bedrock inference + the blocking
connectivity gate, #253).

## Error handling

- Router hard-rejects any model output outside the closed enum: logged as a
  fault, never repaired, never defaulted, never retried into a nearest match
  ([[adr-15-chatbot-two-tier]] rule 2).
- Unauthenticated request to `/chatui/` or its endpoint: redirect to sign-in,
  no router invocation, no leaked enum contents.
- Router/inference failure: composer shows a generic failure state; the
  response carries `Cache-Control: no-store` as an authenticated,
  personalized response ([[adr-06-cache]] rule 4).

## Shadow-test spec

A real browser, driven over the Chrome DevTools Protocol, replays: sign in →
submit a matching utterance → observe the action descriptor render; sign in
→ submit a non-matching utterance → observe NO_MATCH; sign in as a user
without the gated group → submit the utterance that would match the gated
intent for a privileged user → observe NO_MATCH/ESCALATE, never the gated
action; visit `/chatui/` signed out → observe the sign-in redirect. Until
this project's shadow-test runner exists, this entry may reach `building`,
never `shipped` ([[BDD]]).
